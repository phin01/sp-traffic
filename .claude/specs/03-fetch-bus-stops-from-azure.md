# Implementation Specification: Fetch Bus Stops from Azure Blob Storage and Insert into PostgreSQL

## Overview

This specification defines how to retrieve bus stop data from Azure Blob Storage (generated by `previsao_staging` script) and insert it into PostgreSQL for downstream analytics.

---

## Architecture

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐     ┌──────────────────┐
│  Azure Blob │────▶│   paradas_staging│────▶│    In-Memory   │────▶│ PostgreSQL       │
│   Storage   │     │      main.py     │     │    Data Store  │     │ (stg_bus_stops) │
└─────────────┘     └──────────────────┘     └─────────────────┘     └──────────────────┘
```

**Data Flow:**
1. Script connects to Azure Blob Storage using connection string from `.env`
2. Lists blobs in container, filters for `.json` files
3. Selects most recent file by `last_modified` timestamp
4. Downloads blob content into memory (no local file save)
5. Parses JSON structure: top-level array → each item has `line_id` + `ps` array
6. Extracts all stops from nested `ps` arrays, building records with lowercase keys (`cp`, `np`, `py`, `px`)
7. Inserts into PostgreSQL staging table using parameterized queries
8. Deduplicates by `cp`, keeping oldest record based on `loaded_at` timestamp

## Database Schema

### Table: `staging.stg_bus_stops`

```sql
CREATE TABLE IF NOT EXISTS staging.stg_bus_stops (
    cp integer PRIMARY KEY,              -- Bus stop code (unique identifier)
    np text,                             -- Bus stop name (may be empty)
    py numeric(10, 7),                   -- Y coordinate/latitude
    px numeric(10, 7),                   -- X coordinate/longitude
    line_id text,                        -- Associated bus line ID
    source_file text,                    -- Source filename for audit trail
    loaded_at timestamp with time zone DEFAULT NOW()
);

-- Indexes for query performance
CREATE INDEX idx_stg_bus_stops_line_id ON staging.stg_bus_stops(line_id);
CREATE INDEX idx_stg_bus_stops_py_px ON staging.stg_bus_stops(py, px);
```

**Field definitions:**

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `cp` | integer | PRIMARY KEY | Bus stop code (unique) |
| `np` | text | NULLABLE | Bus stop name |
| `py` | numeric(10, 7) | NOT NULL | Y coordinate/latitude |
| `px` | numeric(10, 7) | NOT NULL | X coordinate/longitude |
| `line_id` | text | NULLABLE | Associated bus line ID |
| `source_file` | text | DEFAULT 'paradas_staging/latest.json' | Source filename for audit trail |
| `loaded_at` | timestamp with time zone | DEFAULT NOW() | When the record was loaded |

## Implementation Steps

### Step 1: Environment Setup & Database Connection

Follow the same pattern as `previsao_staging/main.py`:
- Load `.env` using `find_dotenv()` from `python-dotenv`
- Import SQLAlchemy and read DB credentials from env vars
- Create database engine with `pool_pre_ping=True`

### Step 2: Fetch Latest File from Azure Blob Storage

1. Connect to Azure storage using `BlobServiceClient.from_connection_string(AZURE_STORAGE_KEY)`
2. Get container client for `AZURE_CONTAINER_NAME`
3. List all blobs, filter for `.json` files only
4. If no JSON files found → log warning and exit gracefully
5. Sort by `last_modified` timestamp, select most recent file
6. Download blob content using `download_blob()` method
7. Parse JSON into Python dict/list structure (keep in memory, don't save locally)

### Step 3: Extract Bus Stops from JSON Structure

1. Iterate through top-level array items
2. For each item with valid `ps` key containing an array:
   - Loop through all stops in the `ps` array
   - Build stop record dict with fields: `cp`, `np`, `py`, `px`, `line_id`
3. Collect all stops into a list
4. Log total count extracted (expected ~327)

### Step 4: Insert into PostgreSQL with Deduplication

1. Create table `staging.stg_bus_stops` if not exists using parameterized SQL
2. Insert all stop records in bulk using parameterized queries
3. Remove duplicates by `cp`, keeping oldest record based on `loaded_at`:
   - Use `DELETE ... USING` for efficient set-based operation
   - Delete rows where same `cp` has later `loaded_at` timestamp
4. Count remaining unique stops and log result

## Error Handling Strategy

| Scenario | Action | Retry Policy |
|----------|--------|--------------|
| Azure Blob Storage connection failure | Log error, exit with code 1 | No retry |
| Container not found or inaccessible | Log error, exit with code 1 | No retry |
| No JSON files in container | Log warning, exit gracefully (no crash) | No retry |
| Invalid/missing JSON structure | Log error, exit with code 1 | No retry |
| Database connection failure | Log error, exit with code 1 | No retry |
| Zero bus stops found after extraction | Log warning, abort execution | No retry |

## Performance Considerations

### Memory Management
- JSON data loaded entirely into memory before processing
- Current implementation assumes < 10MB payload (327 stops)
- For larger datasets, consider streaming approach

### Database Performance
- SQLAlchemy connection pool handles batch inserts efficiently
- Deduplication uses `DELETE ... USING` for set-based operation
- Indexes on `line_id` and `(py, px)` support common query patterns
